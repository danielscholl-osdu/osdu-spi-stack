# ADR-010: Azure Key Vault for Secret Management

**Status:** Accepted

## Context

OSDU services need access to Azure PaaS connection metadata: CosmosDB endpoints and keys, Service Bus connection strings, storage account names, tenant/subscription IDs. These values are generated during infrastructure provisioning and must be available to services at runtime.

Options:
1. **Kubernetes Secrets**: Simple but not auditable, no rotation, visible to anyone with cluster access.
2. **Azure Key Vault**: Centralized, auditable, supports rotation, accessed via Workload Identity.
3. **ConfigMap + Workload Identity**: Endpoints in ConfigMap, credentials via token exchange.

## Decision

Use a hybrid approach:
- **Key Vault**: Store all Azure PaaS connection metadata (endpoints, keys, connection strings). Services access these via the Key Vault Secrets Provider CSI driver or the Key Vault SDK with Workload Identity.
- **osdu-config ConfigMap**: Store non-sensitive endpoint configuration (hostnames, ports, partition names) for envFrom injection.
- **Kubernetes Secrets**: Only for in-cluster middleware credentials (Elasticsearch, Redis, PostgreSQL passwords) generated by the CLI.

The CLI populates Key Vault secrets during Phase 3 (IAM) from infrastructure provisioning outputs.

## Consequences

- Sensitive values (CosmosDB keys, Service Bus connection strings) are in Key Vault, not in Kubernetes.
- Key Vault access is audited; every secret read is logged.
- In-cluster secrets are limited to three middleware passwords; these are generated once and stored in a seed secret for idempotent re-creation.
- OSDU services use both envFrom (ConfigMap) and Key Vault SDK for configuration, matching the osdu-spi-infra reference deployment pattern.
